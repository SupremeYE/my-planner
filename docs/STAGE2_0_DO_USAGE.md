## 1. do 시각 실제 사용량 (실데이터)

### 1.1 do 채워진 비율

라이브 `todos` 집계:

| 지표 | 값 | 비고 |
|---|---:|---|
| 전체 todos | **30** | earliest `2026-05-27`, latest `2026-07-18` |
| `do_start` 또는 `do_end` 채워짐 | **22 (73.3%)** | |
| `do_start` **AND** `do_end` 둘 다 | **22 (73.3%)** | do는 항상 쌍으로 채워짐(한쪽만 있는 행 0) |
| `do_elapsed_sec > 0` (타이머 실측) | **4 (13.3%)** | 22개 중 4개만 타이머 기록 보유 |
| `plan_start` 또는 `plan_end` 채워짐 | **9 (30.0%)** | 계획 시각은 오히려 소수 |

> ⚠️ **역설**: do(73%)가 plan(30%)보다 훨씬 많이 채워져 있다. 이 앱에서 시각 기록의 주력은 "계획"이 아니라 "실적"이다.

### 1.2 plan과 do가 다른 행 비율

| 지표 | 값 | 해석 |
|---|---:|---|
| plan·do **둘 다** 범위 있음 | **6** | "계획 vs 실제" 비교가 성립하는 유일한 행들 |
| 그중 plan ≠ do (시각이 실제로 다름) | **4** | 비교에 의미가 있는 행 |
| do만 있고 plan 없음 (`do_only_no_plan`) | **16** | **do 22개 중 16개**가 계획 없이 실적만 존재 |

> **핵심**: do가 채워진 22행 중 **16행(73%)은 plan이 아예 없다.** 즉 대부분의 do는 "계획과 비교하려고" 적은 게 아니라, **타임라인 DO 레인에 실행 결과만 직접 그린 것**이다. "계획 1~3시 vs 실제 4~5시" 식 비교가 성립하는 행은 6개(그중 유의미하게 다른 건 4개)뿐이다.

### 1.3 do가 어떻게 채워졌는지 추적

**모달 do 입력칸으로 채워진 것과 타임라인으로 채워진 것을 구분할 명시적 소스 컬럼은 없다(확인 불가).** 그러나 코드 게이트로 **강한 간접 증거**가 있다:

- 모달 do 입력칸은 `hasDoTime = !!(todo?.doStart || todo?.doEnd)` 게이트로만 노출된다(`TodoModal.tsx:80, 468`). 즉 **이미 do가 있는 기존 할일에만** 입력칸이 뜬다. **신규 할일·계획전용 할일에서는 모달로 do를 만들 수 없다.**
- 따라서 `do_only_no_plan` 16행은 **모달이 만든 게 아니다**(모달은 do를 생성하지 못하므로). 생성 경로는 타임라인 DO 레인 추가(`TimelineAddModal` `initialLane='do'`, `Timeline.tsx:43`)·타임라인 DO 드래그(`Timeline.tsx:223-250`)·스톱워치뿐이다.
- **결론: 모달 do 입력칸은 "생성 경로"가 아니라 "편집 전용" 경로다.** do의 실질 생성은 전부 타임라인/타이머에서 일어난다.

**`todo_time_blocks` / `do_elapsed_sec` / `do_start~do_end` 정합성:**

| 지표 | 값 |
|---|---:|
| `todo_time_blocks` 전체 행 | **0** |
| 블록을 가진 todo 수 | **0** |
| do가 있으나 블록이 없는 todo | **22 (전부)** |

> **`todo_time_blocks`(Stage 3 세션 시스템)는 이 DB에서 한 번도 사용된 적이 없다(0행).** "누적 시간"의 dual-read 폴백(`timeBlocks.ts:27-31`)은 항상 do_* 경로로 떨어진다. desync는 블록↔do_* 사이가 아니라 **`do_elapsed_sec` ↔ `do_start~do_end` 사이**에 실재한다(§3.1).

### 1.4 최근 추세 (기능이 살아있나)

| 구간 | todos | do 채워진 수 | 비율 |
|---|---:|---:|---:|
| 최근 30일 | 20 | 14 | **70%** |
| 최근 90일(=전체) | 30 | 22 | 73% |
| 90일 이전 | 0 | 0 | — |

> 전체 데이터가 최근 ~50일 안에 몰려 있어 장기 증감은 확인 불가. 다만 **최근 30일에도 70%가 do를 기록** → do 기능은 **죽어가는 게 아니라 현재도 주력으로 살아 있다.** (단, "살아 있는 do"의 대부분은 모달이 아니라 타임라인에서 나온다 — §1.3.)

## 2. "계획 vs 실제" 비교가 이미 있는 곳

### 2.1 타임리포트 페이지 — **없음**

- 태그/프로젝트별 누적 집계 헬퍼 `elapsedByKey`(`timeBlocks.ts:54`)가 "시간 리포트용"으로 정의·테스트되어 있으나 **어떤 컴포넌트에서도 호출되지 않는다**(grep 결과 `timeBlocks.test.ts`에서만 사용). **전용 타임리포트 페이지는 존재하지 않는다.**

### 2.2 회고/리뷰 페이지 — **비교 안 함**

- `ReviewsView.tsx`에 `doStart`/`planStart`/`실적`/`계획` 문자열 **0건**. 리뷰는 plan vs do를 다루지 않는다.

### 2.3 타임라인 자체 — **여기가 유일한 시각적 비교**

- `Timeline.tsx`의 탭은 `'plan' | 'do' | 'compare'`이며 **기본값이 `'compare'`**(`Timeline.tsx:100`).
- `compare` 모드는 `getTimelineLaneBounds`(`:735-743`)로 화면을 **좌 PLAN 레인 / 우 DO 레인으로 분할 렌더** → 한 항목의 계획 블록과 실적 블록이 **나란히 시각적으로 대비**된다.
- do 초과 감지(`isDoOvertimeVsPlan`, `todoDoDuration.ts:49`)로 DO가 PLAN보다 길면 초과 스타일(`OVERTIME_BAR_*`)까지 표시 → **"계획보다 오래 걸림"을 타임라인이 이미 시각화**한다.

### 2.4 대시보드 — 비교 아님(합계만)

- `DashboardView.tsx`는 `dateDoSeconds`로 오늘/어제 **DO 총 초**만 표시(`:463, :473`). plan과 대비하지 않는 단순 실행량 합계다.

### 2.5 "계획 vs 실제"를 볼 수 있는 곳 — 전체 목록

| 위치 | 비교 성격 | 근거 |
|---|---|---|
| **타임라인 `compare` 탭** (일간·주간) | ✅ 항목별 PLAN\|DO 레인 나란히 + 초과 강조 | `Timeline.tsx:100, 735-743`, `todoDoDuration.ts:49` |
| 대시보드 | △ DO 총량만(비교 아님) | `DashboardView.tsx:463,473` |
| 리뷰/회고 | ✗ 없음 | `ReviewsView.tsx` grep 0건 |
| 타임리포트 페이지 | ✗ 페이지 자체가 없음 | `elapsedByKey` 미사용 |
| **모달 do 입력칸** | △ 숫자 2쌍을 위아래로 편집(시각적 대비 아님) | `TodoModal.tsx:468-483` |

> **결론: 항목 단위의 "계획 vs 실제" 시각 비교는 타임라인 compare 탭이 이미, 그리고 유일하게 제공한다.** 모달의 do 입력칸은 그 비교를 **중복하지 않으며**(나란히 대비하지 않고 숫자만 편집), 오히려 편집 전용 보조 경로에 가깝다.

## 3. 모달 do 입력칸의 실제 동작

### 3.1 do 입력칸 ↔ 누적 시간 관계 (desync 재확인)

- 모달 "실적 시작/실적 종료"는 로컬 state `doStart`/`doEnd`를 편집(`TodoModal.tsx:77-78, 473-479`) → 저장 시 `buildChanges`가 `hasDoTime`일 때만 `do_start/do_end`에 반영(`:265`).
- "누적 시간" = `totalElapsedForTodo(timeBlocks, todo)`(`:117`). 블록이 0행이므로 항상 `todoDoDurationSeconds(todo)`로 폴백.
- **`todoDoDurationSeconds`는 `do_elapsed_sec`를 `do_start~do_end`보다 우선한다**(`todoDoDuration.ts:6`).

**⇒ desync 실재 (재확인됨):**
- **`do_elapsed_sec`가 있는 4행**: 모달에서 do_start/do_end를 바꿔도 **누적 시간은 그대로**다(타이머 초가 우선). 사용자가 "실적 종료"를 고쳐도 누적 시간 숫자가 안 변함 → 혼란.
- **나머지 18행**(do는 있으나 타이머 없음): do_start/do_end가 곧 누적 시간의 소스이므로 모달 편집이 **정상 반영**된다.
- 추가로, 모달의 `accumulated`는 **prop `todo`에만 의존**(`:120`)하고 로컬 `doStart/doEnd` state를 보지 않는다 → 저장 전에는 편집 중 값이 누적 시간에 즉시 미리보기되지 않는다(저장·Realtime 후 재계산).

### 3.2 모달 do 값 ↔ 타임라인 DO 레인 반영

- **양방향**이며 do_start/do_end 컬럼을 공유한다:
  - 모달 편집 → `do_start/do_end` 저장 → 타임라인 DO 막대는 이 컬럼을 읽어 렌더(`Timeline.tsx:857-871, 885-887`).
  - 타임라인 DO 드래그 → 같은 `do_start/do_end` 갱신(`Timeline.tsx:236-243`).
- 단, **타이머(do_elapsed_sec) 축은 별개**다. 타임라인 초과 표시·누적 시간은 elapsed를 우선하므로, 모달의 do_start/do_end만 고치면 그 두 곳과 어긋날 수 있다(§3.1).

### 3.3 `hasDoTime` 게이트(`TodoModal.tsx:80`)의 정확한 동작

- `hasDoTime = !!(todo?.doStart || todo?.doEnd)` — **모달 열림 시점의 todo prop 기준.**
- true일 때만:
  1. "실적 시작/실적 종료" 입력칸 UI 노출(`:468`),
  2. 계획 라벨이 `시작/종료` → `계획 시작/계획 종료`로 전환(`:440, 451`),
  3. `buildChanges`가 do_*를 저장에 포함(`:265`).
- false면 do 관련 UI·저장이 전부 비활성 → **모달로는 do를 새로 만들 수 없다(편집 전용).**

## 4. 제거 시 영향

### 4.1 모달 do 입력칸 제거 시 깨지는 것

- **do 생성 경로는 안 깨진다**: do는 원래 모달로 못 만든다(§3.3). 생성은 타임라인 DO 레인 추가·DO 드래그·스톱워치가 담당 → 그대로 유지.
- **do 편집(수정) 경로는 축소된다**:
  - PC: 타임라인 DO 블록 드래그/리사이즈로 수정 가능(단 **15분 스냅** — 정밀 수정 불가, `BLOCK_SNAP_MIN`).
  - 모바일: 타임라인 DO 블록 **롱프레스** 드래그로 수정 가능(정밀·손쉬움 낮음).
  - **타임라인이 없는 화면(`/todos`·`/inbox`·`/weekly`·`/projects`)에서 여는 모달**에서는 do를 볼·고칠 방법이 사라진다. (단 이 화면들은 원래 do 기록 용도가 아니다.)
- **타이머 기록(do_elapsed_sec) 4행**: 모달 do칸을 없애도 손실 없음(그 값은 애초에 do칸이 아니라 스톱워치가 씀).

### 4.2 "읽기전용 요약"으로 강등 시 표시 값의 소스

- **진실 소스 = `do_start`/`do_end`** (블록 0행이므로 timeBlocks는 무의미). 요약 표기는 두 가지 선택:
  - do 구간 그대로 `HH:mm ~ HH:mm` (16행+이 이 소스로 정합).
  - 소요 시간은 `todoDoDurationSeconds`(elapsed 우선) — 단 이러면 표기 시각(do_start~do_end)과 소요(elapsed)가 4행에서 어긋날 수 있으니 **한 소스로 통일 필요**(예: do_start~do_end만 표기).

### 4.3 기존 do를 읽기전용으로만 보여줄 때 — 수정 수요

- plan≠do로 실제 어긋난 행은 **4개**뿐. 이 값들을 사후 **정정할 실수요는 낮다**(소량·과거 기록).
- 정밀 수정이 필요한 케이스가 생기면 타임라인 드래그(15분 스냅)로 커버되나, **분 단위 정밀 정정은 불가**해진다. 순수 읽기전용이면 이 정밀 정정 경로가 없어진다 → **드문 수요이지만 0은 아님.**

## 5. 산출물 — 결론 요약

### do 사용량 요약
- **채워짐**: 30개 중 **22개(73%)** 가 do 보유. 계획(plan)은 30%에 불과 — 이 앱의 시각 기록 주력은 실적이다.
- **plan과 다른 비율**: do 22개 중 **16개(73%)는 plan이 아예 없음**(비교 대상 없음). plan·do 둘 다 있는 행은 6개, 그중 유의미하게 다른 건 **4개**.
- **증감 추세**: 최근 30일에도 do 70% 기록 → 기능은 **현재도 살아 있음**(단 do의 생성은 타임라인/타이머에서 나오지, 모달에서 나오지 않음).

### 비교 UI 현황
- 항목별 "계획 vs 실제" **시각 비교는 타임라인 `compare` 탭(기본값)이 이미 유일하게 제공**(좌 PLAN / 우 DO + 초과 강조).
- 리뷰·전용 타임리포트 페이지에는 비교 **없음**. 대시보드는 DO 총량만.
- **모달 do 입력칸은 이 비교를 중복하지 않는다**(숫자 편집일 뿐, 나란한 대비 아님).

### desync 재확인
- `todo_time_blocks`는 **0행**(미사용). desync는 **`do_elapsed_sec`(타이머) ↔ `do_start~do_end`(모달·드래그)** 사이에 실재.
- **타이머 기록이 있는 4행**: 모달에서 do 시각을 고쳐도 **누적 시간은 안 바뀐다**(elapsed 우선). → 모달 do 편집이 사용자에게 오해를 준다.
- 나머지 18행은 모달 편집이 누적 시간·타임라인에 정상 반영.

### 권고 근거 (결정은 사용자 몫)

- **(a) 완전 제거를 지지하는 근거**
  - 비교는 타임라인 compare 탭에 이미 있음(§2.3, 2.5).
  - 모달 do칸은 **생성 경로가 아니라 편집 전용**이고, do 생성/주력 기록은 전부 타임라인/타이머에서 일어남(§1.3, 3.3).
  - do 자체는 타임라인 드래그·스톱워치로 계속 입력/수정 가능 → 제거해도 데이터 흐름 안 깨짐(§4.1).
  - 사용자 직관("모달에 왜 있는지 모르겠다")과 데이터가 일치.

- **(b) 읽기전용 요약을 지지하는 근거** ← **데이터가 가장 강하게 지지**
  - 모달 do 편집은 **타이머 4행에서 desync를 일으켜 오해를 준다**(§3.1) → 편집은 없애되,
  - 그래도 모달을 여는 순간 "이 할일 실제로 몇 시에 했지"를 **보여줄 값(do_start~do_end)은 존재**(22행) → 정보로서 표시 가치 있음.
  - 즉 "편집은 desync 위험 + 타임라인에 이미 있음 → 편집 제거 / 표시는 유지"가 데이터와 가장 정합.

- **(c) 유지를 지지하는 근거 (약함)**
  - 타임라인이 없는 화면에서 do를 **분 단위로 정밀 정정**할 유일 경로가 모달이라는 점(§4.3) — 단 실제 정정 수요 행은 4개로 매우 적음.

> **정리**: 데이터는 **(b) 읽기전용 요약**을 가장 강하게 지지한다(비교·생성은 타임라인이 담당, 모달 편집은 desync 위험). **(a) 완전 제거**도 방어 가능(정밀 정정 수요가 4행으로 미미). **(c) 유지**의 근거는 가장 약하다. 표시 값을 넣는다면 소스는 `do_start~do_end` 단일로 통일할 것(§4.2).

### 제거(또는 읽기전용화) 시 깨지는 것 목록
1. `/todos`·`/inbox`·`/weekly`·`/projects` 등 **타임라인 없는 화면에서 여는 모달**의 do 분 단위 정정 경로 상실(읽기전용화·제거 공통). — 영향 행 ≤ 4, 미미.
2. 타임라인 드래그는 **15분 스냅**이라, 분 단위 정밀 do 수정은 모달 제거 시 불가능해짐(정밀 수정 수요가 있을 때만 문제).
3. 그 외 데이터 흐름(생성·타임라인 표시·타이머·누적)은 **깨지지 않음** — do 생성/렌더/집계가 모두 모달과 독립(§4.1).
4. (읽기전용화 시 추가 주의) 표시 소스를 `do_elapsed_sec` 우선으로 잡으면 표기 시각과 소요가 4행에서 어긋남 → **`do_start~do_end` 단일 소스로 표기**해야 함(§4.2).
